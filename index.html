Let's Talk About Dating

So I'm here to talk about dating. [pause] ...JavaScript date libraries. Get your minds out of the gutter.

We Need to Talk


// JavaScript's built-in Date is... complicated
new Date('2026-02-14')    // Feb 13 or 14? Depends on timezone!
new Date(2026, 1, 14)     // Month is 0-indexed. Of course.
new Date('02/14/2026')    // Works in Chrome, breaks in Safari

// It's not you, it's Date.
Like any bad relationship, the native Date object has red flags. Zero-indexed months, timezone surprises, no built-in formatting. We deserve better.

Let me introduce tonight's contestants...

moment.js

"We all dated this one first"

The High School Sweetheart

  • Status: It's complicated (deprecated)
  • Size: ~300KB — "a lot of baggage"
  • Red flags: Mutable, no tree-shaking, maintenance mode

import moment from 'moment'; // 300KB. Hope you're worth it.

const valentines = moment('2026-02-14');
valentines.add(1, 'day');   // Mutates! The original is gone forever
valentines.format('MMMM Do YYYY'); // "February 15th 2026"

// moment taught us what we want. But it's time to move on.
Moment was everyone's first date library. Great API, taught us what good date manipulation looks like. But it's mutable, huge, and the maintainers themselves said "please stop using us for new projects."

date-fns

"Functional, independent, drama-free"

The One Your Friends Recommend

  • Status: Open to commitment (tree-shakeable)
  • Size: Import only what you need — ~2-10KB typical
  • Green flags: Pure functions, immutable, TypeScript native

import { format, addDays } from 'date-fns'; // Just what you need

const valentines = new Date(2026, 1, 14);
const dayAfter = addDays(valentines, 1);  // New date! Original untouched
format(valentines, 'MMMM do, yyyy');      // "February 14th, 2026"

// 200+ functions. Only pay for what you use.
date-fns is the functional programming answer to dates. No wrapper object, just pure functions that take and return native Date objects. Your bundler loves it because tree-shaking actually works.

Day.js

"Same great personality, 1/50th the baggage"

The Low-Maintenance One

  • Status: Ready to mingle (2KB)
  • Size: 2KB minified — "travels light"
  • Green flags: moment-compatible API, immutable, plugin system

import dayjs from 'dayjs'; // 2KB. That's it. That's the bundle.

const valentines = dayjs('2026-02-14');
const dayAfter = valentines.add(1, 'day'); // Immutable! Returns new
valentines.format('MMMM D, YYYY');         // "February 14, 2026"

// If moment was your ex, Day.js is the upgrade.
Day.js is basically the answer to "what if moment but tiny and immutable?" Almost identical API. 2 kilobytes. If you're migrating from moment, this is the least painful path.

Luxon

"Built by moment's creators. They learned from their mistakes."

The Sophisticated Rebound

  • Status: Committed (immutable by design)
  • Size: ~23KB — "carries weight, but it's all muscle"
  • Green flags: Built-in timezone & Intl support, by the moment team

import { DateTime } from 'luxon'; // The redemption arc

const valentines = DateTime.fromISO('2026-02-14');
const dayAfter = valentines.plus({ days: 1 }); // Immutable
valentines.toFormat('MMMM dd, yyyy');           // "February 14, 2026"

// Native Intl support. No locale files to import.
valentines.setLocale('fr').toFormat('MMMM dd'); // "février 14"
Luxon was built by the moment team as their "we know better now" project. Immutable, relies on native Intl APIs for locales so no extra bundle weight for internationalization.

Temporal API

"The one you're waiting for"

The Soulmate (Still Moving to Town)

  • Status: It's complicated (Stage 3 proposal, not in all browsers yet)
  • Size: 0KB — it's built into the language
  • Green flags: Immutable, timezone-native, no library needed

// No import. It's just... there.

const valentines = Temporal.PlainDate.from('2026-02-14');
const dayAfter = valentines.add({ days: 1 }); // Immutable, of course
valentines.toLocaleString('en-US', { dateStyle: 'long' });
// "February 14, 2026"

// Separate types for dates, times, and datetimes. Finally.
const time = Temporal.PlainTime.from('19:30');
const zoned = Temporal.ZonedDateTime.from('2026-02-14T19:30[America/Chicago]');
Temporal is the endgame. It's a TC39 proposal (Stage 3) that builds everything the library ecosystem taught us directly into JavaScript. Immutable, separate types for dates vs times vs zoned datetimes, first-class timezone support. The catch? It's not in all browsers yet. Polyfills exist but are heavy. Keep your eye on this one.

Who Should You Date?

Migrating from moment? → Day.js (familiar API, easy switch)
Bundle size is everything? → date-fns (tree-shake to just what you need)
Heavy timezone / i18n needs? → Luxon (built-in Intl, no extras)
Starting fresh? → date-fns or Day.js (both great modern choices)
Playing the long game? → Temporal (the future, but not ready yet)
moment.js? → We need to have a talk...
There's no single "best" library — it depends on your use case. But there IS a worst choice for new projects, and it rhymes with "schmo-ment."
